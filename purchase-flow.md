# Purchase Flow

## რას ნიშნავს ეს ტერმინები

### `success page`
ეს არის გვერდი, რომელსაც მყიდველი ხედავს გადახდის დასრულებისთანავე.
აქ უნდა ჩანდეს:
- შეკვეთის ნომერი
- რომელი წიგნი იყიდა
- თანხა
- რა მოხდება შემდეგ

### `email`
ეს არის ავტომატური წერილი, რომელიც იგზავნება მყიდველის ელფოსტაზე წარმატებული გადახდის შემდეგ.
ამ წერილში ჩვეულებრივ არის:
- წიგნის სახელი
- შეკვეთის ნომერი
- გადმოწერის ღილაკი
- მხარდაჭერის კონტაქტი

### `download link`
ეს არის ბმული, რომლითაც მყიდველი წიგნს იწერს.
სწორი ვარიანტი უნდა იყოს:
- დროებით მოქმედი
- შეზღუდული ჩამოტვირთვების რაოდენობით
- კონკრეტულ შეკვეთასთან მიბმული

## სწორი რეალური flow

1. მყიდველი ავსებს purchase ფორმას.
2. payment gateway ამოწმებს და ადასტურებს გადახდას.
3. webhook ან edge function ამოწმებს რომ გადახდა მართლა წარმატებულია.
4. `sales` ცხრილში შეკვეთის სტატუსი იცვლება `paid`-ზე.
5. იქმნება `download token`.
6. buyer email-ზე იგზავნება წერილი `download link`-ით.
7. success page-ზე ჩანს:
   - შეკვეთა მიღებულია
   - ელფოსტა გაგზავნილია
   - საჭიროების შემთხვევაში resend ღილაკი
8. როცა მყიდველი დააჭერს ბმულს, backend ამოწმებს token-ს და აძლევს დროებით signed URL-ს წიგნის ფაილზე.

## რატომ არ უნდა იყოს პირდაპირ public ebook link

თუ ebook ფაილი პირდაპირ public ბმულით გაიცემა:
- ყველამ შეიძლება გაუზიაროს
- გადახდასთან აღარ იქნება მიბმული
- ფაილი თავისუფლად გავრცელდება

ამიტომ:
- ყდა შეიძლება იყოს public
- ebook ფაილი ჯობს დარჩეს private bucket-ში
- ჩამოტვირთვა უნდა მოხდეს signed URL-ით ან token-ით

## ამ პროექტში რა არის უკვე

- purchase form
- success page
- შეკვეთის ჩაწერის flow
- სატესტო fallback purchase

## რა აკლია რეალურ გაყიდვამდე

- payment gateway
- webhook / edge function
- email sender
- download token validation
- signed download URL

## რა არის შემდეგი ყველაზე სწორი ნაბიჯი

1. ავირჩიოთ payment gateway
2. დავამატოთ Supabase Edge Function
3. Edge Function-მა:
   - გადაამოწმოს გადახდა
   - შექმნას token
   - გააგზავნოს email
   - განაახლოს `sales.status`
